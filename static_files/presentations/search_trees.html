PART 01 · BST 搜尋成本

同一組 key，可以有不同的搜尋成本

BST 的 inorder 順序固定，但樹形不固定。若每個 key 被搜尋的機率相同，平均比較次數只看深度；若機率不同，常被查到的 key 應該靠近 root。

›按「單步」比較兩棵 BST 的搜尋成本。
Expected comparisons
等機率左樹2.4
等機率右樹2.2
加權左樹1.85
加權右樹2.05
這裡不是單純追求「最矮」：如果搜尋機率不同，高機率 key 的深度會主導平均成本。
Extended Binary Tree 在原本 BST 的每個空 child 補上 external node，可以把「搜尋失敗」也算進成本。若 $I$ 是 internal path length、$E$ 是 external path length、$n$ 是 internal nodes 數，則每個 internal node 會讓 external path 總長多出 2，因此 $E=I+2n$。
PART 02 · Optimal Binary Search Tree

Optimal BST：用 DP 選 root

給定成功搜尋機率 $p_i$ 與失敗搜尋機率 $q_i$，Optimal BST 要找出總期望成本最低的 BST。核心想法是：每個區間都試著選一個 root，左右子樹再用最佳解。

›固定例題：a = 10, 15, 20, 25。
Recurrence
w[i,j] = q[i] + Σ(k=i+1..j)(p[k]+q[k]) c[i,i] = 0 c[i,j] = w[i,j] + min(c[i,l-1] + c[l,j]) r[i,j] = argmin l T[0,n] reconstructs from r[0,n]
若直接對每個區間試所有 root，區間數是 $O(n^2)$，每格最多試 $O(n)$ 個 root，所以基本 DP 是 $O(n^3)$。
PART 03 · AVL 快速回顧

AVL：用 balance factor 維持高度

AVL 是 height-balanced BST。每個 node 記錄 $BF(v)=h_L-h_R$，合法值只允許 $-1,0,1$。插入造成最近祖先 $A$ 的 balance factor 變成 $\pm2$ 時，用 LL/RR/LR/RL 旋轉修復。

›選一個 AVL case，觀察旋轉前後 inorder 不變。
One sentence

AVL 的重點不是「旋轉很神奇」，而是旋轉只改局部 parent/child 關係，不改 inorder sequence，因此仍然是 BST。

search$O(\log n)$
insert/delete$O(\log n)$
額外資訊BF/height
PART 04 · Red-Black Tree

Red-Black：用顏色放寬 AVL 的平衡

Red-black tree 是 extended BST。它不要求左右高度差最多 1，而是用紅黑規則限制最長路徑不會超過最短黑高路徑的兩倍，因此高度仍是 $O(\log n)$。

›固定插入序列：50, 10, 80, 90, 70, 60, 65, 62。
Rules and height

根與 external nodes 是 black；不能有連續 red；所有 root-to-external path 的 black 數相同。

每條路徑長度介於 $r$ 與 $2r$ 之間，所以 $h \le 2\log_2(n+1)$。因此 search、insert、delete 都是 $O(\log n)$。
PART 05 · Splay Tree（補充）

Splay：每次操作後把目標伸展到 root

Splay tree 不存 balance factor 或顏色。它允許單次操作最壞 $O(n)$，但靠著每次搜尋/插入/刪除後 splay 到 root，達成 amortized $O(\log n)$。

›目標 key 5 會透過 zig-zig / zig-zag 類型旋轉往上。
Why useful

如果某些資料近期常被查詢，splay 會自然把它們拉近 root，具有 locality 的效果。

單次最壞$O(n)$
攤銷成本$O(\log n)$
額外資訊無
PART 06 · m-way Search Tree

一個 node 裡可以放多個 key

m-way search tree 的 node 不是只放一個 key，而是放 $K_1,\dots,K_n$ 與 $A_0,\dots,A_n$ 子樹指標。搜尋時先在 node 內找 key 的區間，再往對應 child 走。

›查找 28：先比較 root 的 20, 40，再進入中間子樹。
Routing rule
在 node 內找 i，使 K_i ≤ x < K_{i+1} 若 x 等於某個 K_i：成功 否則往 A_i 走
分支數變大後，高度會下降。這是 B-tree 適合磁碟/資料庫索引的基礎。
PART 07 · B-tree

B-tree：保持每個 node 半滿，所有葉子同層

order $m$ 的 B-tree 是有容量限制的 m-way search tree。除了 root 以外，每個內部 node 至少有 $\lceil m/2\rceil$ 個 children，最多 $m$ 個 children，所有 external nodes 在同一層。

›切換插入與刪除情境，觀察 split、borrow、merge。
Height bound

若有 $N$ 個 keys、order 為 $m$、高度為 $h$，分支數至少約為 $\lceil m/2\rceil$，所以高度是 $O(\log_m N)$ 等級。

PPT 的例子：order $m=200$、$N \le 2\times10^6-2$ 時，高度可以不超過 3。這解釋了為什麼資料庫索引很在意「一次讀很多 key」。
PART 08 · B+ Tree

B+ Tree：資料都在 leaf，leaf 串成 linked list

B+ tree 的 internal nodes 只當 index，真正資料放在 leaf/data nodes。所有 leaf 用 linked list 串起來，所以範圍查詢不用一直回到 root。

›範圍查詢：先走 index 找到第一個 leaf，再沿 leaf links 掃描。
B-tree vs B+ tree
B-treekeys/records 可在內部
B+ treerecords 在 leaves
range queryleaf links
B+ tree 犧牲一點搜尋路徑上的直接命中，換來更規律的 leaf storage 與高效率 range scan。
PART 09 · 總覽比較

每種搜尋樹解決的是不同瓶頸

不要把所有 balanced tree 都記成「某種旋轉」。先問：瓶頸是高度？搜尋機率？局部性？磁碟 I/O？範圍查詢？

結構核心想法典型成本適合場景
普通 BST左小右大平均可好，最壞 $O(n)$概念基礎、小型資料
Optimal BST依搜尋機率安排 root建構 DP $O(n^3)$已知查詢分布且資料較靜態
AVL嚴格高度平衡search/insert/delete $O(\log n)$搜尋很多、想要低高度
Red-Black用紅黑規則放寬平衡search/insert/delete $O(\log n)$標準 library map/set 常用
Splay操作後把目標拉到 rootamortized $O(\log n)$有 locality 的查詢序列
B-tree高分支、node 半滿、葉同層$O(\log N)$，高度很小磁碟與資料庫索引
B+ tree資料在 leaf，leaf 串接搜尋 $O(\log N)$，range scan 快資料庫 range query、檔案系統索引
