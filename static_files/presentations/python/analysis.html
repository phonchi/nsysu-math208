PROLOGUE · 開場

演算法分析：比「誰的程式快」更聰明的問法 pythonds §2.1–2.2

同一個問題有很多寫法。「哪個好？」不能只用碼表量：機器快慢、資料大小都會干擾。 演算法分析問的是：當輸入規模 n 變大，工作量怎麼成長？ 我們數「基本操作次數」T(n)，只留下成長最快的主導項。

›1 加到 n：迴圈法 T(n)=n 次加法 vs 高斯公式 3 個運算。按按鈕比較「操作次數」。
兩種解法 CODE
# 迴圈版：加 n 次 def sum_of_n(n): the_sum = 0 for i in range(1, n + 1): the_sum = the_sum + i return the_sum # 高斯公式版：三個運算，n 再大也一樣 the_sum = (n * (n + 1)) // 2
怎麼在 Python 計時？
import time start = time.time() # 夾住要量的程式碼 # ... 要計時的程式 ... elapsed = time.time() - start # 更講究：timeit.Timer 重複跑 1000 次
講義的實測：迴圈版 n 從 10⁵ 加到 10⁶，時間差不多也乘十；公式版對 10⁵ 到 10⁸ 四種 n 的耗時幾乎是同一個數字。 講義還做了 list 對 dict 的 in 比賽：list 隨 n 線性變慢、dict 平坦如常。 短小的操作要重複跑很多次再取平均，雜訊才壓得下來。
benchmark 的干擾因素
CPU、直譯器的快取與垃圾回收、 當下的系統負載，都會影響碼表讀數。操作次數的成長級別才是演算法本身的性質， 這就是接下來 Big-O 要抓的東西。
PART 01 · 記號

Big-O：只留主導項 pythonds §2.3

T(n) = 5n² + 27n + 1005 → 當 n 夠大，n² 說了算：O(n²)。 常數與低次項在成長率面前都是雜訊。

f(n)名稱n=10n=1,000n=10⁶典型例子
1常數111索引 a_list[i]、dict 查詢（平均）
log n對數31020binary search
n線性101,00010⁶走訪、sequential search
n log n線性對數3310⁴2×10⁷merge sort
n²平方10010⁶10¹²雙層迴圈、bubble sort
2ⁿ指數1,02410³⁰¹—天真費波那契、河內塔

逐行數一次 T(n)

拿一段真的程式來數。三個賦值（3）、外圈跑 n 次的內圈各貢獻 n 次（3n²… 這裡照講義的例子）： T(n) = 3 + 3n² + 2n + 1 = 3n² + 2n + 4。 n 一大，3n² 說了算，其他項連同係數 3 全部丟掉：O(n²)。 數的時候不用精確到每一行，抓住「哪一段被執行最多次」就夠了。

最好、最差、平均

有些演算法的表現不只看 n，還看資料內容。在陣列裡找一個值： 運氣好第一格就中（best case，O(1)）、運氣差找到最後一格或根本不在（worst case，O(n)）、 平均攤下來要看一半（average case）。講 Big-O 時預設講 worst case，因為它是保證； 第 5 章的雜湊表會是「平均 O(1)、最差 O(n)」這種分開報的典型例子。

PART 03 · 案例研究

Anagram 偵測：同一題的四種成長率 pythonds §2.4

「earth 和 heart 是不是變位詞？」四種解法都對，但成長率天差地遠。 這是「正確 ≠ 好」的最佳教材。

›用解法 4（Count and Compare）檢查 earth vs heart。
四解法總覽 CODE
# 解法 1 Checking Off：對 s1 每個字元，去 s2 找到就劃掉 → O(n²) # 解法 2 Sort and Compare：兩字串排序後逐字比 → O(n log n) # 解法 3 Brute Force：列出 s1 全部排列 → O(n!)，免談 # 解法 4 Count and Compare：數 26 個字母出現次數 → O(n) def anagram_solution_4(s1, s2): c1 = [0] * 26; c2 = [0] * 26 for ch in s1: c1[ord(ch) - ord("a")] += 1 for ch in s2: c2[ord(ch) - ord("a")] += 1 return c1 == c2
空間換時間
解法 4 用兩個 26 格計數陣列換到 O(n)。 時間與空間的交易是演算法設計最常用的槓桿之一。

四個解法攤開看

解法 1：Checking Off，O(n²) CODE
def anagram_solution_1(s1, s2): still_ok = (len(s1) == len(s2)) a_list = list(s2) for 每個 s1 的字元（still_ok 時）: 在 a_list 裡線性找它 # 內圈：O(n) if found: del a_list[pos_2] # 劃掉 else: still_ok = False return still_ok # n 個字元 × 每個找 O(n) = O(n²)
完整版在講義 deck02。比較次數是 1+2+…+n = n(n+1)/2，主導項 n²/2。
解法 2：Sort and Compare，O(n log n) CODE
def anagram_solution_2(s1, s2): a_list_1 = list(s1); a_list_1.sort() a_list_2 = list(s2); a_list_2.sort() # 排序後逐字比對，一個不同就 False return a_list_1 逐字等於 a_list_2 # 迴圈 O(n)，但 sort 是 O(n log n)：貴的那步說了算
解法 3：Brute Force，O(n!)
列出 s1 全部排列、看 s2 在不在裡面。 20 個字元的排列數是 20! ≈ 2.4×10¹⁸：就算一微秒生一個，也要七萬多年。 正確、可寫、完全不可用，這種選項的存在是為了提醒你先估量級再動手。
解法 4：Count and Compare，O(n) CODE
def anagram_solution_4(s1, s2): c1 = [0] * 26 # 26 個字母的計數器 c2 = [0] * 26 for ch in s1: c1[ord(ch) - ord("a")] += 1 for ch in s2: c2[ord(ch) - ord("a")] += 1 return c1 == c2 # T(n) = 2n + 26 → O(n)

PART 04 · Python 容器

list 的操作成本：為什麼 append 是攤銷 O(1) pythonds §2.5–2.6

list 底層是連續陣列。索引 O(1)； append 平常 O(1)，偶爾裝滿要搬到更大的家：把搬家成本平均到每次操作，仍是 O(1)（攤銷）。 但 insert(0, x) 每次都要整體右移 → O(n)。 （下面的動畫用「容量翻倍」示意；CPython 實際的過度配置比例更保守，攤銷 O(1) 的道理相同。）

›連按 append 觀察容量成長與搬家。
list 成本表（背這張）
a_list[i] 讀寫O(1)
append / pop()O(1) 攤銷
insert(0) / pop(0) / del（前端或中間）O(n)
走訪 / in（無序）O(n)
切片 a_list[a:b]（k 個元素）O(k)
sortO(n log n)
四種把 0..999 收進 list 的寫法 CODE
test1: l = l + [i] # 串接：每次造新 list，最慢 test2: l.append(i) # 攤銷 O(1) test3: [i for i in range(1000)] # 生成式，更快 test4: list(range(1000)) # 內建一次到位，最快
講義用 timeit.Timer 對四種寫法各跑 1000 次：串接版慢一個數量級以上（總量是 O(n²)），後三種同級遞快。
量出 Big-O 的土方法
把 n 加倍再量一次，看時間變幾倍： 接近 1 倍是 O(1)、2 倍是 O(n)、4 倍是 O(n²)。講義用這招驗證了 pop() 平坦不動、pop(0) 隨 n 線性爬升， 還用 big-o 套件對量測數據自動判定最接近的成長級別。
PART 05 · 雜湊容器

dict 與 set：平均 O(1) 的代價與條件 pythonds §2.7

dict（雜湊表）讓「查 key」平均 O(1)：不用排序、不用走訪。 細節（hash 函數、碰撞、載入因子）第 5 章展開；本章先記住成本表與「平均」兩個字的份量。

操作list（無序）list（已排序）dict / set
查找 contains(x)O(n)O(log n)O(1) 平均／O(n) 最差
插入O(1) 尾端O(n)（挪位）O(1) 平均
刪除O(n)O(n)O(1) 平均
依大小序輸出需先排序O(n)—（保插入序，非大小序）
›按按鈕模擬「在 10 萬元素中查 1000 次」的操作次數比較。
怎麼選容器？
要次序（排名、範圍查詢）→ 已排序 list 搭二分搜尋。
只要快速存在性/對應 → set／dict。
小資料（< 100）→ 什麼都很快，選最簡單的。
